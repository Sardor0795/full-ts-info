TypeScript Basic Types

TypeScript adds a type layer on top of JavaScript. Every value has a type, and TypeScript can check that you're using values correctly at compile time — catching bugs before your code ever runs.

Primitive types are the building blocks. You annotate variables with a colon followed by the type: 

let name: string = "Alice";
let age: number = 30;
let isActive: boolean = true;

// null and undefined are their own types
let nothing: null = null;
let notDefined: undefined = undefined;

any vs unknown — Both can hold any value, but unknown is the safe version. With any, TypeScript gives up checking entirely. With unknown, you must narrow the type before using it: 

// any — no type checking (avoid this!)
let risky: any = "hello";
risky.toFixed(2); // No error, but crashes at runtime!

// unknown — safe: must check before using
let safe: unknown = "hello";
// safe.toFixed(2);  // Error! Can't use unknown directly

if (typeof safe === "string") {
  console.log(safe.toUpperCase()); // OK — narrowed to string
}

void and never — void means a function doesn't return a value. never means something that should never happen: 

function greet(name: string): void {
  console.log(`Hello, ${name}`);
}

function throwError(msg: string): never {
  throw new Error(msg); // never returns
}

Key Takeaways

  • string, number, boolean are the three main primitives
  • Avoid any — use unknown when the type isn't known, then narrow it
  • void = returns nothing, never = can never happen (throws or infinite loop)
  • null and undefined are separate types — enable strictNullChecks in tsconfig

Type Inference

TypeScript is smart — it can figure out types automatically based on the value you assign. You don't have to annotate everything. This is called type inference. 

// TS infers the type from the assigned value
let message = "hello";      // inferred as string
let count = 42;              // inferred as number
let active = true;           // inferred as boolean
let items = [1, 2, 3];       // inferred as number[]

// Return type is inferred too
function add(a: number, b: number) {
  return a + b; // return type inferred as number
}

When should you annotate explicitly? There are key situations where adding types manually is better: 

// 1. Function parameters — always annotate
function greet(name: string) { ... }

// 2. When initializing without a value
let result: string;
result = fetchName();

// 3. When the inferred type is too wide
let status: "loading" | "done" | "error" = "loading";
// Without annotation, TS would infer just `string`

// 4. Complex return types for public APIs
function getUser(id: number): User | undefined { ... }

Contextual typing — TypeScript infers types from context, like callbacks: 

const names = ["Alice", "Bob"];
// TS knows `name` is a string from the array context
names.forEach((name) => {
  console.log(name.toUpperCase()); // no annotation needed
});

// Event handlers also get contextual types
document.addEventListener("click", (event) => {
  console.log(event.clientX); // event is MouseEvent
});

Key Takeaways

  • Let TS infer when the type is obvious from the assigned value
  • Always annotate function parameters — TS can't infer those
  • Annotate when you need a narrower type than what TS would infer
  • Contextual typing works in callbacks, event handlers, and array methods
  • Less annotations = cleaner code, but explicit is better when there's ambiguity

Interfaces & Type Aliases

Both interface and type let you define the shape of an object. They're similar but have a few key differences. 

// Interface — defines an object shape
interface User {
  id: number;
  name: string;
  email: string;
}

// Type alias — can do the same thing
type Product = {
  id: number;
  title: string;
  price: number;
};

Optional and readonly fields:

interface User {
  id: number;
  name: string;
  email?: string;             // optional — may be undefined
  readonly createdAt: Date;   // can't be changed after creation
}

const user: User = { id: 1, name: "Alice", createdAt: new Date() };
// user.createdAt = new Date();  // Error! readonly
// user.email is string | undefined

Extending / composing:

// Interface extends interface
interface Admin extends User {
  role: "admin" | "superadmin";
  permissions: string[];
}

// Type uses intersection (&)
type Admin = User & {
  role: "admin" | "superadmin";
  permissions: string[];
};

When to use which? Use interface for object shapes (especially public APIs and libraries — they support declaration merging). Use type when you need unions, intersections, mapped types, or non-object types: 

// Only `type` can do these:
type ID = string | number;
type Pair = [string, number];
type Callback = (data: string) => void;

Key Takeaways

  • interface — best for object shapes; supports extends and declaration merging
  • type — more flexible; can represent unions, tuples, primitives, and mapped types
  • Use ? for optional properties, readonly for immutable ones
  • Interfaces can be extended; types use & intersection for composition
  • Convention: use interface for objects, type for everything else

Union & Literal Types

Union types let a value be one of several types, using the | operator. This is one of TypeScript's most powerful features. 

// A value can be either string or number
let id: string | number;
id = "abc";  // OK
id = 123;    // OK
// id = true; // Error!

// You must narrow before using type-specific methods
function printId(id: string | number) {
  if (typeof id === "string") {
    console.log(id.toUpperCase()); // string methods
  } else {
    console.log(id.toFixed(2));     // number methods
  }
}

Literal types restrict a value to specific exact values: 

type Direction = "up" | "down" | "left" | "right";
type DiceRoll = 1 | 2 | 3 | 4 | 5 | 6;

function move(dir: Direction) { ... }
move("up");     // OK
// move("diagonal"); // Error! not in the union

Intersection types combine multiple types into one using &: 

type HasName = { name: string };
type HasAge = { age: number };
type Person = HasName & HasAge;

const person: Person = {
  name: "Alice",  // must have both
  age: 30         // must have both
};

Discriminated unions — the most useful pattern. Add a shared literal field to tell types apart: 

type Success = { status: "ok"; data: string };
type Failure = { status: "error"; error: string };
type Result = Success | Failure;

function handle(result: Result) {
  if (result.status === "ok") {
    console.log(result.data);   // TS knows it's Success
  } else {
    console.log(result.error);  // TS knows it's Failure
  }
}

Key Takeaways

  • | (union) = "this OR that", & (intersection) = "this AND that"
  • Literal types restrict values to exact strings, numbers, or booleans
  • Always narrow union types before using type-specific operations
  • Discriminated unions (shared literal field) are a must-know pattern

Typing Functions

Functions are the core of any program. TypeScript lets you type parameters, return values, and even define multiple signatures. 

// Basic typed function
function add(a: number, b: number): number {
  return a + b;
}

// Arrow function with types
const multiply = (a: number, b: number): number => a * b;

// Optional and default parameters
function greet(name: string, greeting?: string): string {
  return `${greeting ?? "Hello"}, ${name}!`;
}
greet("Alice");            // "Hello, Alice!"
greet("Alice", "Hi");      // "Hi, Alice!"

Function type expressions — define the shape of a function as a type: 

// Type for a callback function
type MathOp = (a: number, b: number) => number;

const divide: MathOp = (a, b) => a / b;

// As a parameter
function calculate(a: number, b: number, op: MathOp): number {
  return op(a, b);
}

Rest parameters and overloads:

// Rest parameters
function sum(...nums: number[]): number {
  return nums.reduce((a, b) => a + b, 0);
}

// Function overloads — multiple signatures
function format(value: string): string;
function format(value: number): string;
function format(value: string | number): string {
  if (typeof value === "string") return value.trim();
  return value.toFixed(2);
}

Key Takeaways

  • Always annotate function parameters; return types can usually be inferred
  • Use ? for optional params, = value for defaults
  • void = function returns nothing; don't confuse with undefined
  • Function type expressions: (params) => returnType
  • Overloads let you define multiple call signatures for a single function

Arrays, Tuples & Enums

Arrays can be typed two ways. Both are equivalent: 

const nums: number[] = [1, 2, 3];
const names: Array<string> = ["a", "b"];

// Mixed arrays use union types
const mixed: (string | number)[] = [1, "two", 3];

// Readonly arrays — can't push, pop, or modify
const frozen: readonly number[] = [1, 2, 3];
// frozen.push(4);  // Error!

Tuples are fixed-length arrays where each position has a specific type: 

// A tuple: exactly [string, number]
let pair: [string, number] = ["age", 30];

// Named tuples (TS 4.0+) for clarity
type Point = [x: number, y: number];
const origin: Point = [0, 0];

// Optional tuple elements
type HttpResult = [number, string?];
const ok: HttpResult = [200, "OK"];
const err: HttpResult = [500];

Enums define a set of named constants. Prefer const enum or literal unions in most cases: 

// Numeric enum
enum Direction { Up, Down, Left, Right }
let dir: Direction = Direction.Up; // value is 0

// String enum — more readable at runtime
enum Status {
  Active = "ACTIVE",
  Inactive = "INACTIVE",
  Pending = "PENDING"
}

// const enum — inlined at compile time, no runtime object
const enum Color { Red, Green, Blue }

// Modern alternative: literal union (often preferred)
type Status = "active" | "inactive" | "pending";

as const — freezes values to their literal types: 

const ROLES = ["admin", "user", "guest"] as const;
// type: readonly ["admin", "user", "guest"]

type Role = (typeof ROLES)[number]; // "admin" | "user" | "guest"

Key Takeaways

  • T[] or Array<T> for typed arrays; use readonly to prevent mutation
  • Tuples have fixed length and per-position types — great for function returns
  • Prefer string literal unions over enums for most cases
  • as const narrows array/object values to their literal types

Type Assertions

Sometimes you know more about a value's type than TypeScript does. Type assertions let you tell TS to treat a value as a specific type. They don't change the value at runtime — they only affect type checking. 

// as keyword — assert a more specific type
const input = document.getElementById("email") as HTMLInputElement;
input.value = "hello@example.com"; // OK — TS knows it's an input

// Without assertion, TS only knows it's HTMLElement | null
const el = document.getElementById("email");
// el.value; // Error! HTMLElement has no .value

// Double assertion — escape hatch (use very rarely!)
const x = ("hello" as unknown) as number; // compiles, but wrong!

Non-null assertion (!) — tells TS a value is definitely not null or undefined: 

// You know this element exists in your HTML
const app = document.getElementById("app")!; // HTMLElement (not null)

// Useful with Map.get() when you know the key exists
const map = new Map<string, number>();
map.set("count", 42);
const count = map.get("count")!; // number (not undefined)

// ⚠️ Use sparingly — if you're wrong, you get a runtime error

The satisfies operator (TS 4.9+) — validates a value matches a type without widening it: 

type Color = "red" | "green" | "blue";
type Theme = Record<string, Color | Color[]>;

// Without satisfies — TS widens types, loses specifics
const theme1: Theme = {
  primary: "red",
  secondary: ["green", "blue"],
};
// theme1.primary is Color | Color[] — too wide!

// With satisfies — validates AND preserves narrow types
const theme2 = {
  primary: "red",
  secondary: ["green", "blue"],
} satisfies Theme;
// theme2.primary is "red" — exact literal type preserved!
// theme2.secondary is ("green" | "blue")[] — array preserved!

const assertions — freeze values to their narrowest literal types: 

// Without as const — types are widened
const config1 = { api: "/users", method: "GET" };
// { api: string; method: string }

// With as const — narrowed to literal types, deeply readonly
const config2 = { api: "/users", method: "GET" } as const;
// { readonly api: "/users"; readonly method: "GET" }

// Combine satisfies + as const for best of both worlds
const routes = {
  home: "/",
  about: "/about",
  users: "/users",
} as const satisfies Record<string, string>;
// Validated as Record AND preserves literal types

Key Takeaways

  • as Type tells TS to treat a value as a specific type — use when you know more than TS
  • ! (non-null assertion) removes null/undefined — use sparingly, can hide bugs
  • satisfies validates a type match without widening — best of both worlds
  • as const freezes values to literal types — essential for config objects
  • Assertions are compile-time only — they don't change runtime behavior

Generics

Generics let you write reusable code that works with any type while still keeping type safety. Think of them as type parameters — placeholders that get filled in when used. 

// Without generics — you'd need separate functions
function firstNumber(arr: number[]): number { return arr[0]; }
function firstString(arr: string[]): string { return arr[0]; }

// With generics — one function handles all types
function first<T>(arr: T[]): T | undefined {
  return arr[0];
}

first([1, 2, 3]);         // T = number, returns number
first(["a", "b"]);        // T = string, returns string
first<boolean>([true]);  // explicit: T = boolean

Generic constraints limit what types can be used: 

// T must have a .length property
function longest<T extends { length: number }>(a: T, b: T): T {
  return a.length >= b.length ? a : b;
}

longest("hello", "hi");  // OK — strings have .length
longest([1,2], [1]);     // OK — arrays have .length
// longest(10, 20);        // Error! numbers have no .length

Generic interfaces and types:

// Generic interface for API responses
interface ApiResponse<T> {
  data: T;
  status: number;
  message: string;
}

const userRes: ApiResponse<User> = {
  data: { id: 1, name: "Alice" },
  status: 200,
  message: "OK"
};

// Multiple type parameters
function map<T, U>(arr: T[], fn: (item: T) => U): U[] {
  return arr.map(fn);
}

// Default type parameters
interface Container<T = string> {
  value: T;
}

Key Takeaways

  • Generics = type parameters; they preserve type info across a function/class
  • Use extends to constrain what types are accepted
  • TS infers generic types from arguments — you rarely need to specify them
  • Common patterns: ApiResponse<T>, Array<T>, Promise<T>
  • Use multiple type params (<T, U>) when input/output types differ
Type narrowing
typeof, instanceof, in, discriminated unions
▼

Type Narrowing

When you have a union type, TypeScript needs you to narrow it down before you can use type-specific features. TS tracks these checks through control flow analysis. 

// typeof — for primitives
function process(value: string | number) {
  if (typeof value === "string") {
    return value.toUpperCase(); // string
  }
  return value.toFixed(2);       // number
}

// instanceof — for classes
function getDate(value: Date | string) {
  if (value instanceof Date) {
    return value.getTime(); // Date
  }
  return Date.parse(value);  // string
}

The in operator checks if a property exists on an object: 

type Fish = { swim: () => void };
type Bird = { fly: () => void };

function move(animal: Fish | Bird) {
  if ("swim" in animal) {
    animal.swim(); // narrowed to Fish
  } else {
    animal.fly();  // narrowed to Bird
  }
}

Discriminated unions — the most robust narrowing pattern: 

type Shape =
  | { kind: "circle"; radius: number }
  | { kind: "rect"; width: number; height: number };

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2;
    case "rect":
      return shape.width * shape.height;
  }
}

Truthiness narrowing eliminates null/undefined: 

function print(value: string | null | undefined) {
  if (value) {
    console.log(value.toUpperCase()); // string
  }
}

Key Takeaways

  • typeof for primitives, instanceof for classes, in for property checks
  • Discriminated unions + switch = the gold standard for narrowing
  • TS tracks narrowing through if/else, switch, ternaries, and early returns
  • Truthiness checks narrow out null and undefined

Utility Types

TypeScript provides built-in utility types that transform existing types. These are essential tools you'll use daily. 

interface User {
  id: number;
  name: string;
  email: string;
  age: number;
}

// Partial — all properties become optional
type UpdateUser = Partial<User>;
// { id?: number; name?: string; email?: string; age?: number }

// Required — all properties become required
type FullUser = Required<User>;

// Pick — select specific properties
type UserPreview = Pick<User, "id" | "name">;
// { id: number; name: string }

// Omit — remove specific properties
type CreateUser = Omit<User, "id">;
// { name: string; email: string; age: number }

Record, Readonly, and more:

// Record — define an object with typed keys and values
type Scores = Record<string, number>;
const scores: Scores = { math: 95, english: 88 };

// Readonly — all properties become readonly
type FrozenUser = Readonly<User>;

// ReturnType — extract return type of a function
function getUser() {
  return { id: 1, name: "Alice" };
}
type UserResult = ReturnType<typeof getUser>;
// { id: number; name: string }

// Exclude & Extract — filter union members
type Status = "active" | "inactive" | "banned";
type ActiveStatus = Exclude<Status, "banned">;
// "active" | "inactive"

Real-world example — combining utilities:

// API: create requires everything except id, update allows partial
type CreateUserDTO = Omit<User, "id">;
type UpdateUserDTO = Partial<Omit<User, "id">>;

function createUser(data: CreateUserDTO): User { ... }
function updateUser(id: number, data: UpdateUserDTO): User { ... }

Key Takeaways

  • Partial<T> makes all fields optional — perfect for update operations
  • Pick<T, K> selects fields, Omit<T, K> removes them
  • Record<K, V> creates an object type with specific key and value types
  • ReturnType<T> extracts the return type from a function type
  • Combine utilities: Partial<Omit<T, "id">> is a common pattern

Type Guards

Type guards are functions that tell TypeScript "if this returns true, the value is definitely this type." They use the x is Type return syntax. 

// Custom type guard with `is` keyword
interface Cat { meow(): void; purr(): void; }
interface Dog { bark(): void; fetch(): void; }

function isCat(pet: Cat | Dog): pet is Cat {
  return "meow" in pet;
}

function handlePet(pet: Cat | Dog) {
  if (isCat(pet)) {
    pet.purr();  // TS knows it's Cat
  } else {
    pet.fetch(); // TS knows it's Dog
  }
}

Practical example — validating API responses: 

interface ApiError {
  code: number;
  message: string;
}

interface ApiSuccess<T> {
  data: T;
}

type ApiResult<T> = ApiSuccess<T> | ApiError;

function isError<T>(result: ApiResult<T>): result is ApiError {
  return "code" in result;
}

function handleResult<T>(result: ApiResult<T>) {
  if (isError(result)) {
    console.error(result.message); // ApiError
  } else {
    console.log(result.data);       // ApiSuccess<T>
  }
}

Assertion functions (asserts keyword) throw if the condition fails instead of returning boolean: 

function assertDefined<T>(
  val: T | undefined,
  msg?: string
): asserts val is T {
  if (val === undefined) throw new Error(msg ?? "Value is undefined");
}

const name: string | undefined = getName();
assertDefined(name);
console.log(name.toUpperCase()); // TS knows name is string here

Key Takeaways

  • Type guards use param is Type return type to narrow types
  • They let you encapsulate narrowing logic in reusable functions
  • Assertion functions use asserts param is Type and throw on failure
  • Very useful for API responses, input validation, and runtime checks

Classes in TypeScript

TypeScript adds access modifiers, abstract classes, and interface implementation to JavaScript classes. 

class User {
  // Access modifiers
  public name: string;        // accessible anywhere (default)
  private password: string;   // only inside this class
  protected email: string;   // this class + subclasses
  readonly id: number;       // can't be changed after init

  constructor(id: number, name: string, email: string, pw: string) {
    this.id = id;
    this.name = name;
    this.email = email;
    this.password = pw;
  }
}

// Shorthand: parameter properties
class Product {
  constructor(
    public readonly id: number,
    public name: string,
    private _price: number
  ) {}

  get price(): number { return this._price; }
  set price(v: number) { if (v > 0) this._price = v; }
}

implements — a class can promise to match an interface: 

interface Serializable {
  serialize(): string;
}

class Config implements Serializable {
  constructor(private data: Record<string, string>) {}

  serialize(): string {
    return JSON.stringify(this.data);
  }
}

Abstract classes — base classes that can't be instantiated directly: 

abstract class Shape {
  abstract area(): number;       // must be implemented
  abstract perimeter(): number;  // must be implemented

  describe(): string {              // shared logic
    return `Area: ${this.area()}, Perimeter: ${this.perimeter()}`;
  }
}

class Circle extends Shape {
  constructor(private radius: number) { super(); }
  area() { return Math.PI * this.radius ** 2; }
  perimeter() { return 2 * Math.PI * this.radius; }
}

Key Takeaways

  • public (default), private, protected control access
  • Parameter properties in constructor shorthand: constructor(public name: string)
  • implements ensures a class matches an interface contract
  • abstract classes provide base logic; subclasses must implement abstract methods
  • Use readonly for properties that shouldn't change after initialization

Modules & Declaration Files

TypeScript uses ES modules (import/export) for code organization. Any file with a top-level import or export is treated as a module. 

// utils.ts — named exports
export function formatDate(date: Date): string {
  return date.toISOString();
}

export interface Config {
  apiUrl: string;
  timeout: number;
}

// app.ts — importing
import { formatDate, type Config } from "./utils";

// `type` import ensures it's erased at compile time
import type { Config } from "./utils";

Declaration files (.d.ts) describe the types for JavaScript code that doesn't have TypeScript types built in: 

// global.d.ts — declare global variables
declare const API_URL: string;

// module declaration for a JS library without types
declare module "some-js-lib" {
  export function doSomething(x: string): void;
  export interface Options {
    verbose: boolean;
  }
}

@types packages — the DefinitelyTyped community provides types for popular JS libraries: 

// Install types for a library
// npm install --save-dev @types/lodash
// npm install --save-dev @types/node

// Then just import — types work automatically
import _ from "lodash";
_.chunk([1, 2, 3, 4], 2); // fully typed!

Key Takeaways

  • Use import type for type-only imports — they're erased at compile time
  • .d.ts files describe types without implementation
  • @types/* packages provide community-maintained types for JS libraries
  • declare module lets you type untyped third-party modules
  • Any file with import/export is a module; without, it's a global script

Async & Promises

TypeScript adds full type safety to asynchronous code. Promise<T> is a generic type where T is the type of the resolved value. 

// Promise basics — T is the resolved value type
function fetchUser(id: number): Promise<User> {
  return fetch(`/api/users/${id}`)
    .then(res => res.json());
}

// async/await — return type is always Promise<T>
async function getUser(id: number): Promise<User> {
  const res = await fetch(`/api/users/${id}`);
  return res.json();
}

// The awaited value is unwrapped — no longer a Promise
const user = await getUser(1); // user is User, not Promise<User>

Promise combinators — run multiple async operations with type-safe results: 

// Promise.all — waits for all, preserves types in a tuple
const [users, posts] = await Promise.all([
  fetchUsers(),   // Promise<User[]>
  fetchPosts(),   // Promise<Post[]>
]);
// users: User[], posts: Post[] — types are preserved!

// Promise.allSettled — never rejects, returns status for each
const results = await Promise.allSettled([
  fetchUsers(),
  fetchPosts(),
]);
// Each result is:
//   { status: "fulfilled", value: T }
//   { status: "rejected",  reason: unknown }

// Promise.race — resolves/rejects with the first to settle
const fastest = await Promise.race([
  fetchFromCDN(),
  fetchFromOrigin(),
]);

Error handling in async code:

// catch variable is `unknown` (with useUnknownInCatchVariables)
async function loadData() {
  try {
    const data = await fetchApi<User[]>("/users");
    return data;
  } catch (error) {
    // error is `unknown` — must narrow before using
    if (error instanceof Error) {
      console.error(error.message);
    }
    throw error;
  }
}

// Awaited<T> utility type — unwraps Promise types recursively
type A = Awaited<Promise<string>>;             // string
type B = Awaited<Promise<Promise<number>>>;    // number
type C = Awaited<boolean | Promise<string>>;  // boolean | string

Typing async callbacks and utilities: 

// Typed async function type
type AsyncFn<T> = () => Promise<T>;

// Retry wrapper with typed return
async function retry<T>(
  fn: AsyncFn<T>,
  attempts: number = 3
): Promise<T> {
  for (let i = 0; i < attempts; i++) {
    try { return await fn(); }
    catch (e) { if (i === attempts - 1) throw e; }
  }
  throw new Error("Unreachable");
}

const user = await retry(() => fetchUser(1)); // user is User

Key Takeaways

  • Promise<T> is generic — T is the type of the resolved value
  • async functions always return a Promise; await unwraps it
  • Promise.all preserves individual types as a typed tuple
  • Catch variables are unknown — always narrow with instanceof before using

Mapped & Conditional Types

Mapped types let you transform every property in a type. They iterate over keys using in keyof: 

// Make all properties optional (this is how Partial works!)
type MyPartial<T> = {
  [K in keyof T]?: T[K];
};

// Make all properties readonly
type MyReadonly<T> = {
  readonly [K in keyof T]: T[K];
};

// Transform all values to booleans (for form "touched" state)
type Flags<T> = {
  [K in keyof T]: boolean;
};

interface Form { name: string; email: string; }
type FormTouched = Flags<Form>;
// { name: boolean; email: boolean }

Conditional types choose a type based on a condition, using extends as a type-level "if": 

// Basic conditional type
type IsString<T> = T extends string ? "yes" : "no";

type A = IsString<string>;  // "yes"
type B = IsString<number>;  // "no"

// Extract non-nullable types
type NonNull<T> = T extends null | undefined ? never : T;
type C = NonNull<string | null>;  // string

// Combining: map only string properties to uppercase
type StringKeys<T> = {
  [K in keyof T]: T[K] extends string ? K : never;
}[keyof T];

Key modifier remapping with as (TS 4.1+): 

// Remap keys: add "get" prefix
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K];
};

type UserGetters = Getters<{ name: string; age: number }>;
// { getName: () => string; getAge: () => number }

Key Takeaways

  • Mapped types iterate over keys: { [K in keyof T]: NewType }
  • Conditional types are type-level if/else: T extends U ? X : Y
  • They distribute over unions automatically: NonNull<A | B> = NonNull<A> | NonNull<B>
  • as clause remaps keys during iteration

Template Literal Types & infer

TypeScript can use template literal syntax at the type level to create and manipulate string types. 

// Build string types from other types
type EventName = `${"click" | "focus" | "blur"}Event`;
// "clickEvent" | "focusEvent" | "blurEvent"

// Combine multiple unions — creates all combinations
type Align = `${"top" | "bottom"}-${"left" | "right"}`;
// "top-left" | "top-right" | "bottom-left" | "bottom-right"

// Built-in string manipulation types
type Upper = Uppercase<"hello">;      // "HELLO"
type Lower = Lowercase<"HELLO">;      // "hello"
type Cap   = Capitalize<"hello">;     // "Hello"
type Uncap = Uncapitalize<"Hello">;   // "hello"

The infer keyword extracts parts of a type within a conditional: 

// Extract the return type of a function
type MyReturnType<T> = T extends (...args: any[]) => infer R ? R : never;

// Extract array element type
type ElementOf<T> = T extends (infer E)[] ? E : never;
type X = ElementOf<string[]>;  // string

// Extract from template literals
type ExtractId<T> = T extends `user_${infer Id}` ? Id : never;
type Id = ExtractId<"user_123">; // "123"

// Parse route params
type RouteParams<T> = T extends `${string}:${infer Param}/${infer Rest}`
  ? Param | RouteParams<Rest>
  : T extends `${string}:${infer Param}`
    ? Param
    : never;

Key Takeaways

  • Template literal types create string types using `${...}` syntax
  • Unions in template literals produce all possible combinations
  • infer captures parts of a type inside conditional types
  • Uppercase, Lowercase, Capitalize, Uncapitalize transform string types

keyof, typeof & Indexed Access Types

These operators let you derive types from existing values and types — keeping your code DRY. 

// keyof — produces a union of all keys
interface User {
  id: number;
  name: string;
  email: string;
}

type UserKey = keyof User; // "id" | "name" | "email"

// Type-safe property access
function getValue<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user: User = { id: 1, name: "Alice", email: "a@b.c" };
const name = getValue(user, "name"); // type: string
const id = getValue(user, "id");     // type: number

typeof extracts the type from a value at the type level: 

const config = {
  apiUrl: "https://api.example.com",
  timeout: 5000,
  retries: 3,
};

// Derive type from the actual value
type Config = typeof config;
// { apiUrl: string; timeout: number; retries: number }

// Combine with keyof
type ConfigKey = keyof typeof config;
// "apiUrl" | "timeout" | "retries"

Indexed access types — look up a property's type: 

// Access a single property type
type UserName = User["name"]; // string
type UserId = User["id"];     // number

// Access multiple — produces a union
type NameOrEmail = User["name" | "email"]; // string

// Array element type
const roles = ["admin", "user", "guest"] as const;
type Role = (typeof roles)[number]; // "admin" | "user" | "guest"

Key Takeaways

  • keyof T produces a union of all property name strings
  • typeof value extracts a type from a runtime value
  • T[K] accesses the type of property K on type T
  • Combine these operators for powerful type-safe access patterns
  • keyof typeof obj is the go-to pattern for extracting keys from objects

Variance & Type Compatibility

TypeScript uses structural typing — types are compatible if their shapes match, regardless of their names. This is different from Java/C# which use nominal typing. 

interface Point2D { x: number; y: number; }
interface Point3D { x: number; y: number; z: number; }

// Point3D has all properties of Point2D, so it's assignable
const p3: Point3D = { x: 1, y: 2, z: 3 };
const p2: Point2D = p3; // OK! Point3D has x and y

// This class matches the interface — no `implements` needed
class Coordinate {
  constructor(public x: number, public y: number) {}
}
const point: Point2D = new Coordinate(5, 10); // OK!

Covariance means a subtype can be used where a supertype is expected. Contravariance is the reverse, and it appears in function parameters: 

type Animal = { name: string };
type Dog = { name: string; breed: string };

// Covariant: Dog[] is assignable to Animal[]
const dogs: Dog[] = [{ name: "Rex", breed: "Lab" }];
const animals: Animal[] = dogs; // OK

// Contravariant: function params go the opposite direction
type AnimalHandler = (a: Animal) => void;
type DogHandler = (d: Dog) => void;

// With strictFunctionTypes: DogHandler is NOT assignable to AnimalHandler
// Because the handler might receive any Animal, not just a Dog

Excess property checking only applies to object literals directly assigned: 

interface Options { color: string }

// Direct assignment — excess properties caught!
// const opts: Options = { color: "red", size: 10 }; // Error!

// Via variable — excess properties allowed (structural typing)
const config = { color: "red", size: 10 };
const opts: Options = config; // OK — has `color`

Key Takeaways

  • TypeScript is structurally typed — shape matters, names don't
  • A type with extra properties is assignable to one with fewer (subtype to supertype)
  • Function parameters are contravariant with strictFunctionTypes
  • Excess property checking only catches extra props on direct object literal assignments

Typing React Hooks

type State = { count: number; error: string | null };
type Action =
  | { type: "increment" }
  | { type: "decrement" }
  | { type: "reset"; payload: number };

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case "increment": return { ...state, count: state.count + 1 };
    case "decrement": return { ...state, count: state.count - 1 };
    case "reset":     return { ...state, count: action.payload };
  }
}

  return { data, loading, error } as const;

Typing the Context API

React Context with TypeScript requires careful typing of the context value. The recommended pattern avoids null defaults by using a custom hook with a runtime check. 

import { createContext, useContext, useState } from "react";

// 1. Define context type
interface ThemeContext {
  theme: "light" | "dark";
  toggleTheme: () => void;
}

// 2. Create with null default + assertion
const ThemeCtx = createContext<ThemeContext | null>(null);

// 3. Custom hook with runtime check
function useTheme(): ThemeContext {
  const ctx = useContext(ThemeCtx);
  if (!ctx) throw new Error("useTheme must be used within ThemeProvider");
  return ctx;
}

// 4. Provider component
function ThemeProvider({ children }: { children: React.ReactNode }) {
  const [theme, setTheme] = useState<"light" | "dark">("light");
  const toggleTheme = () => setTheme(t => t === "light" ? "dark" : "light");

  return (
    <ThemeCtx.Provider value={{ theme, toggleTheme }}>
      {children}
    </ThemeCtx.Provider>
  );
}

// 5. Consume in components
function Header() {
  const { theme, toggleTheme } = useTheme(); // fully typed!
  return <button onClick={toggleTheme}>{theme}</button>;
}

Key Takeaways

  • Create context with null default: createContext<T | null>(null)
  • Always wrap useContext in a custom hook with a null check
  • This gives consumers a clean, non-nullable type without extra checks
  • The error thrown helps catch missing Provider bugs immediately

Zod + Form Validation

Zod bridges the gap between TypeScript's compile-time types and runtime validation. You define a schema once and get both validation and a TypeScript type. 

import { z } from "zod";

// Define schema
const UserSchema = z.object({
  name: z.string().min(2, "Name must be at least 2 chars"),
  email: z.string().email("Invalid email"),
  age: z.number().min(18, "Must be 18+").max(120),
  role: z.enum(["admin", "user"]),
});

// Infer the TypeScript type from the schema
type User = z.infer<typeof UserSchema>;
// { name: string; email: string; age: number; role: "admin" | "user" }

// Validate at runtime
const result = UserSchema.safeParse(formData);

  console.log(result.data); // fully typed User
} else {
  console.log(result.error.issues); // validation errors
}

    </form>
  );
}

  • Zod defines schemas that serve as both runtime validators and TS types

  • .safeParse() returns a discriminated union: { success, data } or { success, error }
  • Pairs perfectly with React Hook Form via @hookform/resolvers/zod

// middleware.ts
import { NextRequest, NextResponse } from "next/server";

export function middleware(request: NextRequest) {
  // Check auth token from cookies
  const token = request.cookies.get("session")?.value;

  if (!token && request.nextUrl.pathname.startsWith("/dashboard")) {
    const loginUrl = new URL("/login", request.url);
    loginUrl.searchParams.set("from", request.nextUrl.pathname);
    return NextResponse.redirect(loginUrl);
  }

  // Add custom headers
  const response = NextResponse.next();
  response.headers.set("x-pathname", request.nextUrl.pathname);
  return response;
}

// Matcher config — which routes middleware runs on
export const config = {
  matcher: [
    "/dashboard/:path*",
    "/api/:path*",
    "/((?!_next/static|_next/image|favicon.ico).*)",
  ],
};

Error Handling Patterns

TypeScript enables powerful error handling patterns that go beyond try/catch. The Result type makes errors explicit in the type system. 

// Result type — errors are part of the return type, not thrown
type Result<T, E = Error> =
  | { ok: true; data: T }
  | { ok: false; error: E };

async function fetchUser(id: number): Promise<Result<User>> {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) return { ok: false, error: new Error(`HTTP ${res.status}`) };
    return { ok: true, data: await res.json() };
  } catch (e) {
    return { ok: false, error: e instanceof Error ? e : new Error("Unknown") };
  }
}

// Caller is FORCED to handle the error
const result = await fetchUser(1);
if (!result.ok) {
  console.error(result.error);  // handle error
  return;
}
console.log(result.data.name); // data is User

class NotFoundError extends Error {

  readonly code = "VALIDATION" as const;

}

  readonly code = "UNAUTHORIZED" as const;
}

Exhaustive checking with never:

    case "VALIDATION":   return "Invalid input";
    case "UNAUTHORIZED": return "Not authorized";
    default:

  • Result<T, E> makes errors part of the type — callers must handle them

class QueryBuilder<T> {

    this.filters[key] = value;
    return this;
  }

  limit(n: number): this {
    this._limit = n;
    return this;
  }

  orderBy(field: keyof T): this {
    this._orderBy = field;
    return this;
  }

Type-safe event emitter:

type EventMap = {
  userCreated: { user: User };
  postPublished: { post: Post; author: User };
  error: { message: string; code: number };
};

class TypedEmitter<T extends Record<string, any>> {
  private listeners = new Map<string, Function[]>();

  on<K extends keyof T>(event: K, fn: (payload: T[K]) => void): void {
    const fns = this.listeners.get(event as string) ?? [];
    fns.push(fn);
    this.listeners.set(event as string, fns);
  }

  emit<K extends keyof T>(event: K, payload: T[K]): void {
    this.listeners.get(event as string)?.forEach(fn => fn(payload));
  }
}

const bus = new TypedEmitter<EventMap>();
bus.on("userCreated", ({ user }) => { ... }); // user is typed!
bus.emit("error", { message: "fail", code: 500 }); // payload is typed!
